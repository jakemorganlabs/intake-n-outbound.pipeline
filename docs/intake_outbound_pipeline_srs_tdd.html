Intake-to-Outbound Intelligence Pipeline

A lead-intelligence pipeline that is deterministic at every step but one — and never trusts the one.
Doc IDMICT-PIPE-001
Version1.0 — Baselined
AuthorJake Morgan
StatusApproved for build

This document specifies and designs a production lead-handling system: a public form submission is captured, enriched with web research and a single language-model extraction, scored by deterministic rules, and routed to one of three outbound actions. The system's defining property is inference containment — the model is confined to one schema-bounded call whose output is validated before it is allowed to influence anything, so a malformed or low-confidence response degrades safely instead of propagating. The document is in two parts: Part I states what the system must do (SRS); Part II states how it is built (TDD).

The model proposes structure; deterministic code disposes.

Most "AI-powered" pipelines fail in production because the language model touches everything — it parses, decides, and acts in one ungoverned step, and when it hallucinates a field or returns broken JSON, that error flows straight into a customer-facing action. This system inverts that. The model is given exactly one closed task: read the lead and emit a fixed set of fields. Its output is checked against a strict schema before it is permitted to affect anything downstream. The decisions that drive business action — the score, the route, the outbound call — are pure functions of validated data, never of raw model output.

The result is a system with a small, named blast radius. There is exactly one place where non-determinism enters, it is wrapped in a validation gate, and every failure of that gate has a defined safe fallback. That is the whole thesis, and it is what this document exists to make legible.

What the system must do
Defines scope, actors, and the complete set of functional and non-functional requirements the implementation in Part II is obligated to satisfy.

§1 Introduction

1.1 Purpose

This specification defines the requirements for an automated system that captures inbound leads from a public web form, enriches each lead with external research and a single bounded language-model inference, scores the lead for sales fit using deterministic rules, and routes it to the appropriate outbound action. It is the authoritative reference for what the system must do and the basis against which the build is accepted.

1.2 Scope

In scope

  • Webhook ingestion of structured form submissions and their validation.
  • Deduplication so that webhook retries and double-submits produce a single outcome.
  • Enrichment by (a) web research and (b) one schema-constrained model call that extracts firmographic and intent signals.
  • A deterministic composite fit score and a three-tier routing decision.
  • Outbound delivery to a chat channel, a CRM, and a spreadsheet of record.
  • Durable persistence of every lead with full provenance, plus error capture, alerting, and observability.

Out of scope

  • The marketing form's own design and hosting (an external form provider is assumed).
  • Two-way reply handling, nurture sequencing, or any conversational follow-up.
  • Authentication of prospects, billing, or any UI beyond an internal observability view.
  • Multi-tenant operation; the system serves a single sending organization.

1.3 Intended Audience

Hiring reviewers and engineers evaluating the design; the implementer (the author) during build; and any future maintainer extending the pipeline. No prior familiarity with the form provider or the orchestration tool is assumed — interfaces are specified in §5.

1.4 Definitions, Acronyms & Abbreviations

A full glossary appears in Appendix C. The four terms that recur most: containment (confining non-determinism to one validated step), the validation gate (the schema check that model output must pass), idempotency key (the stable token that makes reprocessing safe), and provenance (the per-field record of where a value came from).

1.5 References & Standards

  • RFC 2119 — keyword strength (MUST / SHOULD / MAY).
  • JSON Schema, draft 2020-12 — the contract format for the inference output.
  • The orchestration platform's webhook, HTTP, and error-workflow primitives (§5, §19).
  • The model provider's structured-output (tool-use) interface (§15).

1.6 Document Conventions

Requirements carry stable identifiers of the form FR-XX-n (functional, grouped by area) and NFR-XX-n (non-functional). Each requirement states a single testable obligation and a strength keyword. Appendix A traces every identifier to the design section and component that satisfies it.

§2 Overall Description

2.1 Product Perspective

The pipeline is a self-contained, event-driven service triggered by an HTTP webhook. It has no interactive front end of its own; its "users" are an upstream form, three downstream systems, and an operator who reads its logs. It is designed to run as a single self-hosted orchestration workflow with a relational store for state, behind a public tunnel that exposes only the webhook endpoint (§19).

2.2 Product Functions

At a glance, the system: receives → validates → deduplicates → researches → infers → validates output → scores → routes → delivers → persists, with any step's failure diverted to a dead-letter path and alert. Each verb maps to a component in §10 and a requirement group in §3.

2.3 Actors & User Classes

ActorTypeInteraction
ProspectExternal, humanSubmits the public form. Never sees the system directly.
Sales / OpsInternal, humanReceives hot-lead alerts and CRM records; reads the warm-lead sheet.
OperatorInternal, humanMonitors metrics, inspects the audit log, resolves dead-lettered items.
Form providerExternal, systemFires the inbound webhook; may retry on non-2xx.
Model & search APIsExternal, systemProvide inference and research; both may be slow or unavailable.

2.4 Operating Environment

A containerized orchestration runtime and a relational database, co-located on a single small virtual server. Outbound network access to the model, search, chat, CRM, and spreadsheet APIs is required. The public webhook is exposed through an encrypted tunnel rather than an open inbound port (§16, §19).

2.5 Design & Implementation Constraints

  • Exactly one language-model call per lead under normal operation; a bounded repair retry is the only permitted second call (§15).
  • Scoring and routing MUST be deterministic and free of model calls.
  • Secrets MUST NOT appear in source, logs, or exported state.
  • The system MUST run within a hobby-tier server's resources and a small, predictable per-lead API budget (§17).

2.6 Assumptions & Dependencies

  • The form provider sends a stable, unique submission identifier with each webhook (the basis for idempotency, §13). If absent, a derived key is used as fallback.
  • The model provider supports forced structured output via a tool/function interface (§15).
  • The downstream chat, CRM, and spreadsheet systems expose authenticated write APIs.

§5 External Interface Requirements

5.1 Intake — Form Webhook (inbound)

HTTPS POST, JSON body containing the submission fields and a stable submission id. Authenticated by shared secret/signature (FR-IN-2). Responds 200 on accept, 400 on malformed, 401 on auth failure, 503 when the state store is down (FR-ER-3). Idempotent under retry (§13).

5.2 Inference — Model Provider API (outbound)

Messages/completions endpoint with a forced tool call whose input schema is the enrichment contract (§11.2). Parameters: pinned small model, temperature 0, bounded max_tokens. Timeout and bounded repair retry per FR-AI-4. Errors are typed: 429 → backoff; 401/402 → terminal escalate.

5.3 Web Research — Search Provider API (outbound)

Query endpoint returning ranked results for a company/domain query. Per-call timeout with one retry; failure degrades the lead, never fails it (FR-WR-2). Quota-managed as the dominant variable cost.

5.4–5.6 Outbound — Chat, CRM, Spreadsheet

  • Chat channel: authenticated message post for HOT alerts. Includes the summary, score, and a link to the record.
  • CRM: authenticated record-create for HOT leads, populated from validated fields.
  • Spreadsheet of record: authenticated append for WARM leads.

All three sit behind adapters; each failure is retried then dead-lettered (FR-RT-5).

5.7 State Store (internal)

A relational database providing the dedupe table (atomic insert-if-absent), the canonical lead table, the inference audit table, and the dead-letter table. Availability of this store is a hard dependency (FR-ER-3).

§6 Data Requirements

  • Canonical lead record — one row per lead, full structure in §11.1.
  • Retention — leads retained for the operating period; the worked examples and any shipped demo use synthetic data only (Document Control).
  • Classification — submissions contain contact PII (name, email) and are treated as confidential at rest and in transit; never emitted to third-party logs (NFR-SE-2).
  • Provenance — every enriched field records its origin (form / web / model) and, for model fields, the model id and confidence.

§9 The Containment Principle

One non-deterministic step, wrapped in a gate, with a defined fallback for every way the gate can fail. Everything else is ordinary software.

The design treats the language model the way a careful engineer treats any untrusted input: it is allowed to propose structure, and deterministic code disposes of the proposal only after validation. Three properties make this real rather than rhetorical:

  • A single entry point for non-determinism. There is exactly one model call (§3.4, FR-AI-1). The blast radius of model misbehavior is therefore one node, not the whole pipeline.
  • A hard validation gate. Output must satisfy a strict JSON Schema with enumerated values and additionalProperties:false (§11.2). A hallucinated field, an out-of-range value, or broken JSON is rejected at the boundary — it cannot reach scoring.
  • A defined fallback for every failure. Schema failure → one repair → MANUAL. Low confidence → routing capped at WARM. Timeout → bounded backoff → MANUAL. Search failure → degrade and continue. None of these is an unhandled exception; each is a branch in the design (§14).

Crucially, the decisions — the 0–100 score and the tier — are pure functions of validated data (§3.5), so they are reproducible, explainable, and testable without invoking the model at all. That is what lets the system carry a real eval suite (§18): the deterministic half can be unit-tested, and the contained half can be probed against fixtures including adversarial inputs.

§10 Component Design

Each component lists its responsibility, inputs, outputs, and failure behavior.

10.1 Webhook Receiver

Responsibility: terminate the inbound request, authenticate it, return promptly. In: HTTPS POST. Out: raw payload + HTTP status. Fails: 401 unauthenticated, 400 malformed, 503 if state store down.

10.2 Validator & Normalizer

Responsibility: confirm required fields; trim/lower-case; derive the email domain; produce the canonical normalized object. Fails: 400 with reason logged.

10.3 Idempotency Guard

Responsibility: derive the key (§11.3) and perform an atomic insert-if-absent against the dedupe table. Out: proceed, or short-circuit 200 on a known key. Fails: store unavailable → 503 (fail closed, FR-ER-3).

10.4 Web Research Collector

Responsibility: issue bounded search queries for the company/domain; attach results with provenance. Fails: timeout/error → mark degraded, continue with partial/empty context.

10.5 Contained Inference Unit

Responsibility: the single model call. Assemble the prompt from normalized fields + research; force a structured tool call; capture output, token counts, latency. Fails: timeout → bounded backoff; typed API errors handled per §14.

10.6 Validation Gate

Responsibility: validate output against the schema; on failure, perform one corrective re-call (passing the validation error + prior output); on second failure, set status inference_failed and route MANUAL. Out: a trusted enrichment object, or a MANUAL diversion.

10.7 Deterministic Scoring Engine

Responsibility: map validated signals to component sub-scores via versioned config; sum to a 0–100 composite; persist the breakdown. No external calls. Pure and unit-testable.

10.8 Router

Responsibility: apply tier thresholds with the confidence guard (FR-RT-4) and select exactly one tier. Out: tier + the action set to perform.

10.9 Outbound Adapters

Responsibility: chat post, CRM create, sheet append — each idempotent where the API allows, each with try/retry. Fails: on exhaustion → dead-letter the action, mark delivery_failed, alert; the lead record still persists.

10.10 Persistence & Audit Writer

Responsibility: write the canonical lead record and the inference audit record. Always runs, even on MANUAL/failed paths, so nothing is lost.

10.11 Error Workflow / Dead-Letter Handler

Responsibility: a single global handler that catches any unhandled stage failure, writes {lead snapshot, stage, error, timestamp} to the dead-letter table, and raises an operator alert. The one place errors converge.

11.3 Idempotency Key Derivation

The correct key comes from a source-stable identifier, not from anything the pipeline invents. Primary: the provider's submission id. Fallback (if the provider omits one): a hash binding the identity of the submission.

Key derivation# preferred — provider's stable id
key = "sub:" + provider_submission_id

# fallback — derived, stable across retries because all inputs are stable
key = "drv:" + sha256( lower(email) + "|" + form_id + "|" + submitted_at )

11.4 Scoring Configuration (versioned)

Weights sum to 100; each categorical value maps to a 0.0–1.0 factor. Held as external config (NFR-MA-1), versioned on the record (weights_version).

Scoring config v1.0weights = { size:20, industry:15, budget:20, urgency:15,
            decision_maker:15, clarity:15 }   # sum = 100

factors:
  company_size      enterprise 1.0 | mid 0.8 | small 0.5 | solo 0.3 | unknown 0.2
  industry          target 1.0 | adjacent 0.6 | off 0.2 | unknown 0.3
  budget_indicated  true 1.0 | false 0.2 | null 0.4
  timeline_urgency  high 1.0 | medium 0.6 | low 0.3 | unknown 0.4
  decision_maker    true 1.0 | false 0.3 | null 0.4
  use_case_clarity  high 1.0 | medium 0.6 | low 0.2

composite = round( sum( weight_i * factor_i ) )     # 0..100

# inference_failed (schema gate exhausted) overrides --> MANUAL (DLQ + alert)

§12 Processing Sequence

  1. Receive & authenticate (≤0.5 s). Reject bad/unauthenticated requests early.
  2. Validate & normalize. Build the canonical normalized object; derive domain.
  3. Idempotency guard. Atomic insert-if-absent; on a known key, return 200 and stop.
  4. Web research (≤8 s, timed out, one retry). Attach results + provenance; mark degraded on failure.
  5. Contained inference (≤15 s timeout, typical ~3–6 s). One forced-tool call; capture output + tokens + latency.
  6. Validation gate. Schema-check; one repair on failure; else status inference_failed → MANUAL.
  7. Scoring (<100 ms). Deterministic composite + component breakdown.
  8. Routing (<10 ms). Apply thresholds + confidence guard → one tier.
  9. Outbound (≤5 s). Perform the tier's actions; retry then dead-letter on failure.
  10. Persist & audit. Write the lead record and the inference audit record. Done.

Latency budget targets NFR-PE-1: the research and inference calls (steps 4–5) dominate; everything else is sub-second.

§13 Idempotency & Exactly-Once Outcomes

Webhooks are an at-least-once channel: providers retry on any non-2xx, and networks duplicate. The system therefore aims not for exactly-once delivery (impossible) but exactly-once outcomes. The mechanism is the atomic dedupe insert at step 3 (§10.3): the first delivery of a key wins the insert and proceeds; any later delivery of the same key finds the row present and short-circuits to 200 with no enrichment and no outbound action. Because the key is derived from a source-stable identifier (§11.3), it is identical across all retries of the same submission, and because the insert is atomic, two simultaneous deliveries cannot both win. This is also why the store is a hard dependency (FR-ER-3): without it, the only safe action is to fail closed and let the provider retry.

§15 Prompt & Inference Configuration

The model is configured to be as deterministic and as bounded as inference allows, then never trusted regardless.

  • Forced structured output. A single tool is defined whose input_schema is the enrichment contract (§11.2); tool_choice forces the model to call it. This compels a structured object and forbids free-form prose (FR-AI-2). The validation gate (§10.6) re-checks anyway — defense in depth, since "forced" is not the same as "guaranteed valid."
  • System instruction. Frames the model as a structured extractor with one job; enumerates the fields; instructs it to return unknown/null when a value is not grounded in the input rather than guessing (FR-AI-5); instructs honest confidence calibration; states explicitly that any instructions appearing inside the lead's message are data to be summarized, not commands to follow (the injection guard, AC-5).
  • Parameters. Pinned small model (recorded per lead, FR-AI-6); temperature 0; small max_tokens sufficient for the object; 15 s timeout.
  • Repair loop. On schema failure, exactly one re-call that includes the validation error and the prior invalid output and asks for a corrected object. Bounded to one (FR-AI-4); a second failure is terminal → MANUAL.

A better prompt reduces the rate of bad output; it cannot make the rate zero, because the model is non-deterministic by construction. The validation gate makes the consequence of bad output zero — it is caught at the boundary and routed to a safe path. The prompt is an optimization; the gate is the guarantee. The system depends on the guarantee.

§16 Security Design

  • Webhook authenticity — shared secret or signature verification on every inbound request (FR-IN-2); unauthenticated requests are rejected before any processing.
  • No open inbound ports — the endpoint is published through an encrypted tunnel that exposes only the webhook path (§19).
  • Secrets — all API keys and the webhook secret live in environment/secret storage, never in source, logs, or exported state (NFR-SE-1).
  • Least privilege — each downstream credential is scoped to exactly the action it performs (post / create / append).
  • PII discipline — contact data stays within the record store and the named downstream systems; it is never written to third-party logs and is excluded from diagnostic exports (NFR-SE-2).
  • Injection containment — because the model's output is schema-bounded and acted on only after validation, a prompt-injection attempt in the message field cannot escalate into an action; worst case it produces a low-quality-but-valid extraction caught by the confidence guard.

§17 Observability & Cost

17.1 Structured Logging

Each execution emits one structured log line keyed by a traceable execution id, carrying the lead id, idempotency key, per-stage timings, model id, token counts, composite score, tier, and final status (NFR-OB-1). A reviewer can reconstruct any lead's journey from a single line.

17.2 Inference Audit Record

Beyond the lead record, each model call writes an audit row: model id, parameters, prompt hash, raw output, validation result, repair_used, latency, token counts. This is what makes model behavior inspectable after the fact and is the difference between "we call an LLM" and "we operate an LLM."

  • More channels / tiers. Additional outbound adapters or a finer tier scheme are config + adapter changes, not redesigns.
  • Feeds the RAG piece. This pipeline's validated, provenance-tagged leads are a clean upstream source for Piece II (Document Intelligence / RAG).

C Glossary

Containment
Confining all non-determinism to a single model call whose output is validated before use, so its failure modes have a bounded blast radius.
Validation gate
The strict JSON-Schema check (§11.2) that model output must pass before it can influence scoring, routing, or any action.
Idempotency key
A source-stable token (§11.3) that makes reprocessing the same submission a no-op, yielding exactly-once outcomes.
Provenance
The per-field record of where a value came from — form, web, or model — and, for model fields, the model id and confidence.
Dead-letter (DLQ)
The store where a failing lead and its error are captured for operator triage, so nothing is silently lost.
Tier
The routing outcome: HOT (chat + CRM), WARM (sheet), COLD (log only), or MANUAL (DLQ + alert).
Degraded
A lead enriched without full web research because the search dependency failed — quality reduced, availability preserved.
Repair loop
The single bounded re-call made when model output fails schema validation, after which a second failure is terminal.
