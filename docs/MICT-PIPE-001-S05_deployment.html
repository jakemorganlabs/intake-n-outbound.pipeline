Session S05 of the MICT-PIPE-001 build. This document defines the goal, scope, deliverables, build sequence, and acceptance criteria for a single working session. It is bounded so the work fits in one focused block and produces artifacts a future session can rely on without revision. The five-session arc — Core, Spine, Adapters, Evals, Deploy — is shown below; each session ends with a clean handoff to the next.

Deployment is where the design's edge claims meet the open internet.

The §19 deployment topology is small and deliberate: a single VPS, the orchestration runtime and Postgres as containers, an encrypted tunnel that publishes only the webhook path, and outbound HTTPS to five named APIs. No open inbound ports. No SSH exposed to the internet. The whole edge surface area is one URL.

This session does the unglamorous work that turns a working system on a laptop into something a stranger could hit at https://intake.jakemorganlabs.dev/webhook and have it behave correctly. The success criterion is a smoke test posted from outside the VPS, ending in a lead row and a Slack ping. Once that passes, Piece I is done.

§1 Goal & Exit Criteria

By the end of this session the following are true. Anything short of this means the session is not done, regardless of how much code was written.

  • curl https://intake.jakemorganlabs.dev/webhook from a laptop on another network → 200 + lead row + tier-appropriate outbound effect.
  • Only the webhook path is publicly reachable; the n8n editor and the DB are not exposed to the internet.
  • Secrets are in env files (or a secret store), not in any committed file or any exported workflow JSON.
  • Postgres is backed up nightly via pg_dump + retention.
  • Redeployment runbook in the README: how to redeploy, how to apply a migration, how to rotate a secret.
  • Eval suite passes against the deployed instance (via a separate EVAL_ENV=prod mode).

§2 Prerequisites

Conditions that must be in place before this session can usefully start.

  • S04 deliverables on main; CI green.
  • Hetzner VPS provisioned, hardened (UFW, SSH key-only, fail2ban).
  • Cloudflare account; Cloudflare Tunnel set up with cloudflared installed on the VPS.
  • Domain (or subdomain) configured at Cloudflare DNS pointing to the tunnel.

§3 Scope

In scope

  • docker-compose.yml for the production stack: n8n + Postgres + a cloudflared sidecar (or host-installed).
  • Image pinning (n8n version, Postgres version) — no latest tags.
  • Volume strategy: Postgres data on a named volume; n8n workflows imported at boot from workflows/.
  • .env.production.example documenting every required variable.
  • Cloudflare Tunnel config exposing only /webhook; other paths return 404 at the edge.
  • Database backup: pg_dump via cron, with at least seven days of retention.
  • Deployment runbook in README: redeploy, migrate, rotate secrets, restore from backup.
  • External smoke test: scripts/smoke_prod.sh hitting the public URL.
  • Production eval mode: EVAL_ENV=prod npm run eval using a clean test webhook path or namespaced dedupe keys.

Out of scope

  • Auto-scaling, horizontal redundancy, blue/green deploys — the §19 topology explicitly chooses one box.
  • Long-term log retention or metrics dashboards — deferred to the capstone (MICT-FIELD-005).
  • Multi-tenant operation — explicitly out of scope per §1.2.

§5 Build Sequence

An ordered set of steps. Each step is small enough to complete and verify before moving to the next; together they cover every deliverable.

  1. Write deploy/docker-compose.yml pinning n8n + Postgres versions explicitly. Test the compose stack locally; confirm the smoke test passes against the local compose stack.
  2. Create deploy/.env.production.example listing every required variable with a one-line description. Author the real .env.production outside the repo.
  3. SSH to the Hetzner VPS. Clone the repo. Place .env.production. Bring up the stack with docker compose up -d. Apply migrations.
  4. Import the n8n workflow JSONs at boot — either via n8n's CLI on container start, or via a one-shot import script invoked by the runbook.
  5. Install cloudflared. Authenticate. Create the tunnel. Configure ingress so only the webhook path is routed; everything else returns 404 at the edge.
  6. Configure DNS at Cloudflare: intake.jakemorganlabs.dev CNAME to the tunnel's hostname.
  7. Set up nightly pg_dump via cron with a seven-day rolling retention. Test by restoring to a scratch DB.
  8. From a laptop on a different network, run scripts/smoke_prod.sh with the Worked Example B payload. Assert the lead row and the Slack ping.
  9. Run EVAL_ENV=prod npm run eval against the live instance. Assert green.
  10. Author docs/runbook.md: redeploy (git pull + compose up), migrate (migration script invocation), rotate secrets (which file, which container restart), restore from backup (the exact pg_restore command).
  11. Set up an external uptime ping (UptimeRobot or Cloudflare itself) on the webhook path with a known no-op payload, alerting to email.

§6 Definition of Done

The session ends when every box below is checked. These are stricter than the goal criteria — they include the housekeeping that makes the work durable.

  • External smoke test from a non-VPS network: 200 + lead row + Slack ping.
  • nmap from outside the VPS shows only Cloudflare's edges; no port directly on the VPS public IP is reachable.
  • n8n editor not reachable from the public internet.
  • Postgres not reachable from the public internet.
  • Production eval run: green.
  • Backup created last night; restore test performed once and documented.
  • Runbook present; a second person could redeploy from it without help.
  • Uptime monitor pinging hourly and configured to alert.

§7 Risks & Pitfalls

The mistakes that, in practice, derail sessions of this kind. Each is paired with a concrete mitigation rather than a general warning.

RiskMitigation
Tunnel ingress too permissiveA wildcard ingress rule exposes the n8n editor or the metrics endpoint by accident. Use explicit per-path rules; default 404 at the edge.
Postgres volume not mountedA misconfigured compose file means the DB is in-container and dies with the container, taking every lead with it. Mount a named volume; restart the stack once and confirm data persists.
Secrets leaking via exported workflowsn8n's exported workflow JSON can occasionally inline credentials. Diff before commit; use credential references only; grep -r the repo for token prefixes before pushing.
No rollback planIf a deploy goes wrong, the only path back is reading the runbook for the first time under stress. Practice a redeploy from scratch once now; time it; record the steps.
Backups untestedAn untested backup is not a backup. Restore at least once to a scratch DB and write down the exact commands in the runbook.
Eval suite not run against productionLocal-only evals don't prove the production wiring. Run the suite against the live instance with namespaced keys; that is the difference between 'I think it works in prod' and 'I know it works in prod.'
